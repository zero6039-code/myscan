# 您的网站被 Google 警告“页面重定向”？别慌，一文读懂并修复

您是否在 Google Search Console 里看到过这样一条提示：**“Page with redirect”**（页面重定向）？它通常只影响一两个页面，例如 `http://www.yourdomain.com`，但如果不处理，会影响搜索引擎对您网站的收录，进而降低自然流量。作为一名 Web 安全顾问，我经常帮助客户排查此类问题。这篇文章将用最直白的语言，带您理解这个问题的本质、如何自己动手验证，以及永久解决它的方法。如果在阅读过程中有任何疑问，或者需要更深入的安全审计，欢迎随时联系我。

## 一、什么是“Page with redirect”？

简单来说，当 Google 的爬虫（Googlebot）访问您的某个网址时，它没有直接看到该页面的内容，而是被引导到了另一个网址。这就好比您去一家商店，门口贴着“已搬迁至新地址”，您得再走一段路才能到达真正的店铺。如果只搬一次还好，但如果搬迁通知指着另一个地址，那个地址又指着另一个地址……您就会晕头转向。Googlebot 也一样，当它经过**两次或更多次跳转**才能看到最终页面时，就会标记该页面为“Page with redirect”，并可能不再索引这个原始网址。

这听起来很技术性，但用一句话概括就是：**您的旧网址没有直接指向最终的家，而是让访客（和爬虫）绕了远路。**

## 二、如何判断自己的网站有这个问题？

您不需要成为程序员，只需跟着下面几个步骤动手试试。最常用的方法是使用一个叫 **curl** 的命令行工具。如果您用的是 Windows 电脑，可以打开命令提示符（按 Win+R，输入 `cmd` 回车）；如果您是 Mac 或 Linux 用户，打开终端即可。

### 2.1 用 curl 追踪“搬家路线”

假设您的网站是 `dewsecure.com`，想要检查 `www` 版本是否正常。请在命令行中输入：

```bash
curl -I -L http://www.dewsecure.com/
```

然后观察输出。重点看**每一段 `HTTP/1.1` 或 `HTTP/2` 后面的状态码**。如果是 `301`，就代表一次永久重定向。我们期望的是：**只出现一次 301，然后紧跟着一个 `200`**，表示已经到达了最终页面。

**一个健康的例子**（一次 301 + 200）：
```
HTTP/1.1 301 Moved Permanently
Location: https://dewsecure.com/

HTTP/2 200
(页面内容)
```

**有问题的例子**（两次 301）：
```
HTTP/1.1 301 Moved Permanently
Location: https://www.dewsecure.com/

HTTP/2 301
Location: https://dewsecure.com/

HTTP/2 200
(页面内容)
```
这里出现了两次 301，Google 就会报错。

### 2.2 直接浏览器查看

您还可以在浏览器中打开 `http://www.dewsecure.com/`，看地址栏最终会跳到哪个网址。如果中间闪烁过不同的域名或协议，就说明可能存在多次跳转。

## 三、为什么会出现多次重定向？

最常见的原因有两个：

1. **Cloudflare 的“始终使用 HTTPS”功能开启了**，它会让所有 HTTP 请求先跳转到 HTTPS 一次；然后您的页面规则又让 `www` 跳转到非 `www`，造成第二次跳转。
2. **多条重定向规则叠加**，比如同时设置了域名跳转和强制 HTTPS，但顺序没有优化，导致链路过长。

对于不懂技术的网站主，您完全不需要自己部署这些规则，只需要知道问题出在哪里，然后请您的技术伙伴或服务商按照下面的方案调整即可。

## 四、如何永久修复（附解决方案）

修复的目标是：**让所有访客和爬虫一步到位，从任何旧地址直接跳转到最终地址**。以 `dewsecure.com` 为例，我们希望：
- `http://www.dewsecure.com` → 直接 301 到 `https://dewsecure.com`
- `https://www.dewsecure.com` → 直接 301 到 `https://dewsecure.com`
- 不再有中间跳转。

如果您使用 Cloudflare，具体操作如下：

1. **关闭“始终使用 HTTPS”**  
   进入 Cloudflare 仪表板 → SSL/TLS → 边缘证书 → 关闭 **“Always Use HTTPS”**。

2. **清理多余的页面规则**  
   进入 规则 → 页面规则，删除所有与 `www` 转发相关的旧规则。

3. **创建两条新规则**（用最简洁的方式实现一步跳转）：
   - **规则一**：`http://www.dewsecure.com/*` → 301 到 `https://dewsecure.com/${1}`
   - **规则二**：`https://www.dewsecure.com/*` → 301 到 `https://dewsecure.com/${1}`

如果您的网站不在 Cloudflare 上，也可通过 Apache、Nginx 等服务器配置文件实现类似逻辑。如果不清楚如何配置，请将本文转给您的开发人员，他们一看便知。

最后，在 Google Search Console 的“页面索引”报告中点击 **验证修复**，Google 会重新抓取并更新状态。

## 五、修复后，您还需要注意什么？

- **检查所有页面**：建议用 curl 或浏览器访问您的首页、内页，确保所有 HTTP/HTTPS、www/non-www 都只有一次跳转。
- **添加规范链接（Canonical）**：在您网页的 `<head>` 区域加入 `<link rel="canonical" href="https://dewsecure.com/">`，这能帮助搜索引擎理解哪个网址是您希望的首选版本。
- **定期查看 Search Console**：养成习惯，每月至少查看一次“索引”和“安全与手动操作”报告，及早发现问题。

## 六、您的网站真的安全吗？

重定向问题只是冰山一角。我在日常安全审计中发现，许多网站还存在以下隐患：
- 未启用 HTTPS 或 HTTPS 配置不当
- 敏感信息泄露（如服务器版本）
- 缺少关键安全头（CSP、HSTS 等）
- 低速或高风险的第三方脚本
